# 翼修通 Agent 架构分析报告

## 执行摘要

翼修通系统当前实现了一个基于 LangGraph 的多 Agent 协同工作流，包含 **11 个专业化 Agent 节点**，通过状态机编排实现从故障输入到工单生成的完整闭环。系统架构清晰、职责分离良好，但仍有显著的优化空间。

---

## 一、当前 Agent 架构概览

### 1.1 工作流拓扑

```
START
  ↓
[Router Agent] ──────────────────────────────────────┐
  ↓                                                   │
[Retrieval Agent - Primary] ─→ 证据不足？            │
  ↓                              ↓ Yes               │
  No                        [Retrieval Agent - Retry]│
  ↓                              ↓                    │
[Diagnosis Agent] ←──────────────┘                   │
  ↓                                                   │
[Traceability Agent]                                 │
  ↓                                                   │
[Confidence Agent] ─→ 置信度低？                     │
  ↓                     ↓ Yes                         │
  No               [Review Agent]                     │
  ↓                     ↓                             │
[WorkOrder Agent - Draft] ←─┘                        │
  ↓                                                   │
[WorkOrder Agent - Validate] ─→ 需要修复？           │
  ↓                              ↓ Yes                │
  No                        [WorkOrder Agent - Repair]│
  ↓                              ↓                    │
[Audit Agent] ←──────────────────┘                   │
  ↓                                                   │
END                                                   │
```

### 1.2 Agent 清单与职责

| Agent 名称 | 节点标识 | 核心职责 | 输入 | 输出 |
|-----------|---------|---------|------|------|
| **Router Agent** | `route` | 场景识别与路由 | 用户请求 | 场景类型、置信度、匹配信号 |
| **Retrieval Agent (Primary)** | `retrieve_primary` | 首次知识检索 | 查询、场景类型 | 证据列表、风险等级 |
| **Retrieval Agent (Retry)** | `retrieve_retry` | 低召回二次检索 | 原查询、场景提示 | 增强证据列表 |
| **Diagnosis Agent** | `diagnose` | 生成诊断建议 | 证据、风险规则 | 可能原因、检查项、处置建议 |
| **Traceability Agent** | `trace` | 建立证据溯源 | 诊断结果、证据 | 建议-证据映射 |
| **Confidence Agent** | `score` | 计算置信度 | 证据、溯源、提供者 | 置信度评分、警告 |
| **Review Agent** | `second_opinion` | 低置信度校正 | 诊断、证据、溯源 | 修正后的诊断 |
| **WorkOrder Agent (Draft)** | `draft` | 生成工单草案 | 诊断、风险、溯源 | 结构化工单 |
| **WorkOrder Agent (Validate)** | `validate` | 工单质量检查 | 工单草案 | 验证结果、问题列表 |
| **WorkOrder Agent (Repair)** | `repair_work_order` | 自动修复工单 | 工单、验证问题 | 修复后的工单 |
| **Audit Agent** | `respond` | 审批策略评估 | 所有状态 | 最终响应、审批原因 |

## 二、各 Agent 详细分析

### 2.1 Router Agent（路由智能体）

**当前实现**：
- 基于关键词匹配和故障码前缀识别场景
- 支持三种场景：故障诊断、工艺偏差、质量检验
- 计算路由置信度（0.52-0.97）

**优点**：
- 逻辑清晰，易于理解和维护
- 支持用户显式指定场景（优先级最高）
- 提供详细的匹配信号和原因说明

**不足**：
- 纯规则匹配，缺少机器学习能力
- 关键词列表需要人工维护
- 对模糊或跨场景问题处理能力弱
- 置信度计算公式较简单

**优化建议**：
1. **引入分类模型**：使用轻量级文本分类模型（如 DistilBERT）提升准确率
2. **多标签支持**：允许一个问题同时属于多个场景
3. **动态关键词学习**：从历史数据中自动提取高频关键词
4. **置信度校准**：基于历史准确率校准置信度评分

### 2.2 Retrieval Agent（检索智能体）

**当前实现**：
- 主检索：使用语义向量检索（top-5）
- 重试检索：低召回时添加场景提示词重新检索（top-8）
- 支持案例记忆（从历史工单中学习）
- 结果合并和去重

**优点**：
- 双重检索策略提高召回率
- 集成案例记忆增强实战能力
- 自动触发重试机制

**不足**：
- 仅使用语义检索，缺少关键词精确匹配
- 没有重排序机制
- 检索评分未经过校准
- 缺少检索结果多样性控制

**优化建议**（已在 Spec 中）：
1. **混合检索**：结合 BM25 关键词检索和语义检索
2. **重排序模型**：使用 BGE-reranker 提升 top-k 质量
3. **检索策略优化**：
   - 故障码精确匹配优先
   - 场景特定的权重配置
   - 多样性惩罚（避免返回过于相似的证据）
4. **检索质量监控**：记录检索命中率和用户反馈

### 2.3 Diagnosis Agent（诊断智能体）

**当前实现**：
- 优先使用 LLM 生成结构化诊断（JSON 格式）
- LLM 失败时降级为文本解析
- 最终兜底使用启发式规则
- 三层降级策略保证鲁棒性

**优点**：
- 多层降级保证系统可用性
- 启发式规则覆盖常见场景
- 文本解析逻辑健壮

**不足**：
- LLM prompt 未经过系统优化
- 缺少少样本学习（Few-shot）示例
- 启发式规则覆盖有限
- 没有利用历史成功案例

**优化建议**：
1. **Prompt 工程优化**：
   - 添加角色设定和任务说明
   - 提供 3-5 个高质量示例（Few-shot）
   - 明确输出格式和约束
2. **思维链（Chain-of-Thought）**：让 LLM 先推理再输出结论
3. **案例检索增强**：检索最相似的历史成功案例作为参考
4. **启发式规则扩展**：从历史数据中挖掘新规则
5. **多模型集成**：使用多个 LLM 投票提升准确性

### 2.4 Traceability Agent（溯源智能体）

**当前实现**：
- 为每条建议关联支持证据
- 计算支持度（strong/moderate/weak）
- 识别缺乏证据支持的建议

**优点**：
- 提供可解释性
- 识别低质量建议
- 支持度分级清晰

**不足**：
- 匹配算法较简单（关键词匹配）
- 没有考虑语义相似度
- 缺少证据充分性评估

**优化建议**：
1. **语义匹配**：使用句子嵌入计算建议与证据的语义相似度
2. **证据充分性评分**：评估证据是否足以支持建议
3. **反向验证**：检查证据是否支持其他矛盾的建议
4. **可视化展示**：在前端展示建议-证据关系图

### 2.5 Confidence Agent（置信度智能体）

**当前实现**：
- 综合评估证据质量、溯源质量、提供者可靠性
- 计算 0-100 的置信度评分
- 划分高/中/低三个等级
- 生成警告信息

**优点**：
- 多维度评估
- 评分范围直观
- 提供可操作的警告

**不足**：
- 权重系数硬编码
- 缺少历史数据校准
- 没有考虑场景差异

**优化建议**：
1. **权重学习**：从历史数据中学习最优权重
2. **场景特定模型**：不同场景使用不同的置信度模型
3. **不确定性量化**：提供置信区间而非单一评分
4. **校准曲线**：确保置信度与实际准确率一致

### 2.6 Review Agent（复核智能体）

**当前实现**：
- 低置信度时触发
- 融合启发式修正意见
- 单次执行限制

**优点**：
- 提供二次校正机会
- 防止低质量输出

**不足**：
- 修正逻辑较简单
- 没有利用外部知识
- 单次限制可能不够

**优化建议**：
1. **多轮对话**：允许多轮修正直到达到阈值
2. **专家系统集成**：调用规则引擎进行验证
3. **对抗性验证**：生成反例测试诊断的鲁棒性
4. **人工介入接口**：低置信度时请求人工专家意见

### 2.7 WorkOrder Agent（工单智能体）

**当前实现**：
- 三个子节点：Draft（起草）、Validate（验证）、Repair（修复）
- 自动生成结构化工单
- 质量检查和自动修复
- 单次修复限制

**优点**：
- 工单结构化程度高
- 自动验证和修复
- 支持多种场景模板

**不足**：
- 模板系统不够灵活
- 修复逻辑覆盖有限
- 缺少优先级智能排序
- 没有预估执行时间

**优化建议**（已在 Spec 中）：
1. **模板系统增强**：支持自定义模板和版本管理
2. **智能字段填充**：从上下文自动提取更多信息
3. **步骤优化**：
   - 智能排序（依赖关系、优先级）
   - 预估执行时间
   - 资源需求标注
4. **修复策略扩展**：支持更多验证规则和修复逻辑

### 2.8 Audit Agent（审计智能体）

**当前实现**：
- 评估是否需要人工审批
- 综合多个因素（风险、证据、置信度、验证状态）
- 生成审批原因列表

**优点**：
- 多维度决策
- 原因可追溯
- 去重逻辑清晰

**不足**：
- 决策规则硬编码
- 缺少优先级排序
- 没有学习能力

**优化建议**：
1. **决策树学习**：从历史审批数据中学习决策规则
2. **风险评分模型**：使用机器学习预测审批通过率
3. **审批路由**：根据问题类型路由到不同审批人
4. **审批时效预测**：预估审批所需时间

## 三、整体架构优化建议

### 3.1 新增 Agent 建议

#### 3.1.1 Knowledge Graph Agent（知识图谱智能体）

**职责**：
- 构建和维护航空领域知识图谱
- 提供实体关系查询
- 支持推理和路径查找

**价值**：
- 增强检索的语义理解
- 发现隐含关系
- 支持复杂查询

**实现优先级**：中

#### 3.1.2 Explanation Agent（解释智能体）

**职责**：
- 生成自然语言解释
- 可视化推理过程
- 回答用户追问

**价值**：
- 提升用户信任
- 教育新手用户
- 支持交互式诊断

**实现优先级**：高

#### 3.1.3 Learning Agent（学习智能体）

**职责**：
- 从用户反馈中学习
- 更新模型和规则
- 发现新模式

**价值**：
- 持续改进系统
- 适应新场景
- 减少人工维护

**实现优先级**：中

#### 3.1.4 Multi-Modal Agent（多模态智能体）

**职责**：
- 处理图片、视频输入
- 识别设备状态
- 提取视觉特征

**价值**：
- 支持现场拍照诊断
- 识别视觉缺陷
- 增强诊断准确性

**实现优先级**：低（比赛后）

### 3.2 工作流优化

#### 3.2.1 并行化执行

**当前问题**：所有节点串行执行，延迟较高

**优化方案**：
```python
# 并行执行独立任务
async def parallel_phase():
    results = await asyncio.gather(
        retrieve_evidence(),
        evaluate_risk(),
        load_case_memory(),
    )
    return merge_results(results)
```

**预期收益**：响应时间减少 30-40%

#### 3.2.2 早停机制

**当前问题**：即使高置信度也执行所有节点

**优化方案**：
- 高置信度（>90）+ 充足证据（>5）→ 跳过 Review Agent
- 简单问题（历史精确匹配）→ 直接返回缓存结果

**预期收益**：简单问题响应时间减少 60%

#### 3.2.3 动态路由

**当前问题**：工作流路径固定

**优化方案**：
- 根据问题复杂度选择不同路径
- 简单问题：Router → Retrieval → Diagnosis → Respond
- 复杂问题：完整流程 + 多轮迭代

**预期收益**：平均响应时间减少 25%

### 3.3 状态管理优化

**当前问题**：
- WorkflowState 包含 20+ 字段，难以维护
- 缺少状态版本控制
- 没有状态持久化

**优化方案**：
1. **状态分层**：
   - Core State（核心状态）
   - Agent State（Agent 私有状态）
   - Metadata（元数据）

2. **状态快照**：
   - 每个节点保存状态快照
   - 支持回滚和重放
   - 便于调试和审计

3. **状态持久化**：
   - 长时间运行的任务持久化到数据库
   - 支持断点续传
   - 支持分布式执行

### 3.4 可观测性增强

**当前实现**：
- 基础的执行轨迹记录
- 进度回调机制

**优化建议**：
1. **结构化日志**：
   - 使用 structlog 记录结构化日志
   - 包含 trace_id、span_id
   - 支持日志聚合和查询

2. **指标监控**：
   - 每个 Agent 的执行时间
   - 成功率、失败率
   - 资源使用情况

3. **分布式追踪**：
   - 集成 OpenTelemetry
   - 可视化调用链
   - 性能瓶颈分析

4. **实时监控面板**：
   - Agent 健康状态
   - 工作流执行统计
   - 异常告警

## 四、性能优化建议

### 4.1 缓存策略

**L1 缓存（内存）**：
- 检索结果缓存（5 分钟）
- 模型推理缓存（相同输入）
- 规则评估缓存

**L2 缓存（Redis）**：
- 历史诊断结果（24 小时）
- 知识库索引（持久化）
- 用户会话状态

**预期收益**：重复查询响应时间减少 80%

### 4.2 模型优化

**当前问题**：
- 每次请求都调用 LLM
- 模型推理时间占比 60-70%

**优化方案**：
1. **模型量化**：使用 INT8 量化减少推理时间
2. **批处理**：合并多个请求批量推理
3. **模型蒸馏**：训练小模型处理简单问题
4. **边缘部署**：常见问题使用本地小模型

**预期收益**：推理时间减少 40-50%

### 4.3 数据库优化

**当前问题**：
- 案例记忆查询较慢
- 缺少索引优化

**优化方案**：
1. **索引优化**：为常用查询字段建立索引
2. **查询优化**：使用 JOIN 替代多次查询
3. **读写分离**：读操作使用只读副本
4. **连接池**：复用数据库连接

**预期收益**：数据库查询时间减少 60%

## 五、可靠性增强

### 5.1 容错机制

**当前实现**：
- 多层降级策略
- 异常捕获和日志记录

**增强建议**：
1. **重试策略**：
   - 指数退避重试
   - 最大重试次数限制
   - 幂等性保证

2. **熔断器**：
   - 检测服务异常
   - 自动熔断和恢复
   - 降级到备用服务

3. **超时控制**：
   - 每个 Agent 设置超时
   - 全局超时控制
   - 超时后优雅降级

### 5.2 数据一致性

**当前问题**：
- 工作流中断后状态丢失
- 缺少事务保证

**优化方案**：
1. **事务支持**：
   - 关键操作使用事务
   - 支持回滚
   - 保证 ACID 特性

2. **幂等性设计**：
   - 所有 Agent 操作幂等
   - 支持重复执行
   - 避免副作用

3. **状态机验证**：
   - 验证状态转换合法性
   - 防止非法状态
   - 自动修复不一致

## 六、安全性增强

### 6.1 输入验证

**当前问题**：
- 缺少输入长度限制
- 没有敏感信息过滤

**优化方案**：
1. **输入校验**：
   - 长度限制（防止 DoS）
   - 格式验证
   - 特殊字符过滤

2. **敏感信息检测**：
   - PII 信息脱敏
   - 敏感词过滤
   - 数据加密存储

### 6.2 权限控制

**当前问题**：
- 缺少细粒度权限控制
- 没有操作审计

**优化方案**：
1. **RBAC 权限**：
   - 角色定义
   - 权限矩阵
   - 动态权限检查

2. **操作审计**：
   - 记录所有操作
   - 审计日志不可篡改
   - 支持审计查询

## 九、总结

### 9.1 当前优势

1. **架构清晰**：11 个 Agent 职责明确，易于理解和维护
2. **鲁棒性强**：多层降级策略保证系统可用性
3. **可扩展性好**：基于 LangGraph 的状态机易于扩展
4. **可追溯性强**：完整的执行轨迹和证据链

### 9.2 主要不足

1. **性能瓶颈**：串行执行导致延迟较高
2. **智能化不足**：过度依赖规则，缺少学习能力
3. **可观测性弱**：缺少完善的监控和调试工具
4. **优化空间大**：检索、诊断、置信度评估都有提升空间

### 9.3 优化优先级

**高优先级**（比赛前必做）：
1. 混合检索 + 重排序
2. Prompt 优化
3. 置信度评估增强
4. 可视化展示

**中优先级**（比赛后优化）：
1. 并行化执行
2. 缓存策略
3. Explanation Agent
4. 监控面板

**低优先级**（长期规划）：
1. 多模态支持
2. 知识图谱
3. 分布式部署
4. 自动化学习
