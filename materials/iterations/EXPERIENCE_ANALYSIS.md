# 翼修通（Yixiutong）项目体验分析报告

> 分析日期：2026-04-05
> 分析维度：行业痛点、易用性、用户体验、工作流程、用户交互友好性、可用性
> 优先级：行业痛点 > 易用性 > 用户体验

## 一、行业痛点分析与优化建议

### 1.1 航空维修行业核心痛点

航空制造与运维场景具有以下独特挑战：

| 痛点 | 描述 | 行业影响 |
|------|------|----------|
| **信息孤岛** | 维修手册（CMM/SRM/AMM）、技术通告、故障记录分散在不同系统 | 检索困难，跨系统对比耗时 |
| **经验传承难** | 资深维修人员经验难以标准化复制 | 人才断层风险 |
| **决策风险高** | 误判可能导致安全事故或重大经济损失 | 责任重大，心理压力大 |
| **合规审计严** | 每一步操作需可追溯、可审计 | 文档工作繁重 |
| **时间压力大** | 停机时间直接关联经济损失 | 响应速度要求高 |

### 1.2 当前系统对行业痛点的覆盖评估

| 痛点 | 现有支持 | 不足之处 | 严重程度 |
|------|---------|---------|----------|
| 信息检索 | BM25词频检索 (`retrieval.py:24-31`) | 非真正语义理解，航空专业术语处理差 | ⚠️ 高 |
| 经验沉淀 | `execution_trace` 仅记录执行过程 | 无诊断历史入库、同类故障推荐 | ⚠️ 高 |
| 决策支持 | 置信度+审批流 | 60分阈值一刀切，不区分故障类型 | ⚠️ 高 |
| 合规审计 | `traceability.py` 简单映射 | 支撑强度分级粗糙（strong/partial/weak） | ⚠️ 中 |
| 响应时效 | 串行LangGraph工作流 | 诊断流程较长，无加速机制 | ⚠️ 中 |

### 1.3 优化建议（行业痛点方向）

#### 🔴 高优先级

**1. 引入专业语义检索模型**

- **现状**：`retrieval.py` 使用 `Counter` 词频 + 余弦相似度，并非真正的向量嵌入
- **问题**：航空维修术语（如 "振动异常"、"温升告警"、"批次漂移"）的语义理解能力有限
- **建议**：
  - 短期：挂载航空领域专业词向量（如基于维修手册预训练的 Embedding）
  - 长期：针对 CMM/SRM/AMM 文档结构微调专用语义模型

```python
# retrieval.py:24-31 当前实现（词频Counter）
def _cosine_similarity(a: Counter[str], b: Counter[str]) -> float:
    keys = set(a) | set(b)
    dot = sum(a.get(key, 0) * b.get(key, 0) for key in keys)
    ...

# 建议改为：使用专业嵌入模型
from sentence_transformers import SentenceTransformer
_model = SentenceTransformer("paraphrase-multilingual-MiniLM-L12-v2")
```

**2. 建立知识沉淀机制**

- **现状**：`execution_trace` 仅用于单次诊断记录，无积累
- **问题**：同类故障无法共享历史诊断经验
- **建议**：
  - 诊断完成后自动入库（`diagnosis_history` 表）
  - 新请求时检索相似历史，辅助参考
  - 支持 "历史相似工单推荐" 功能

**3. 风险自适应阈值**

- **现状**：`confidence.py:85` 硬编码 `requires_human_review = overall < 60 or risk_level == "high"`
- **问题**：所有故障类型统一60分阈值，不区分场景风险
- **建议**：
  - 根据故障类型（E-开头 vs PROC- vs QA-）动态调整阈值
  - 高风险设备（如起落架、发动机）自动升压
  - 低风险场景（如清洁、润滑）可适当降低阈值

#### 🟡 中优先级

**4. 航空维修工单标准化**

- **现状**：`work_orders.py` 生成的工单为通用格式
- **问题**：航空维修工单通常需要特定格式（Zone、操作编号、工具航材列表）
- **建议**：
  - 支持工单模板配置（故障诊断 vs 工艺偏差 vs 质量处置）
  - 增加 `zone`（维修区域）、`tooling`（工具）、`parts`（航材）字段
  - 符合航空标准的编号规则（如 `WO-2026-XXXXX`）

**5. MRB 流程支持**

- **现状**：质量场景仅有通用审批流
- **问题**：航空质量处置常需 MRB（材料审查委员会）流程
- **建议**：
  - 质量场景自动触发 MRB 评估
  - 支持 MRB 升级、NDT（非破坏检测）流程
  - 符合适航法规的闭环记录

## 二、易用性分析与优化建议

### 2.1 当前易用性评估

| 维度 | 现状 | 问题 | 影响 |
|------|------|------|------|
| 首次使用 | 需理解 AI 诊断概念 | 缺乏引导和新手教程 | 新用户上手困难 |
| 输入体验 | 纯文本输入 (`DiagnosisPage.tsx:59-73`) | 故障码、现象描述混在一起 | 用户不知道该填什么 |
| 结果理解 | 置信度+诊断建议 | 证据支撑不直观，溯源困难 | 用户不敢信任结果 |
| 纠错成本 | 驳回后需重填 | 无暂存/草稿机制 | 重复工作量大 |
| 多角色协作 | 通过审批流转 | 状态不透明，通知不及时 | 协作效率低 |

### 2.2 优化建议（易用性方向）

#### 🔴 高优先级

**6. 结构化输入向导**

- **现状**：`InputPanel` 组件使用单一文本框输入
- **问题**：用户不清楚需要提供什么信息
- **建议**：
  - 分离为四个独立字段：`故障码`、`设备类型`、`现象描述`、`上下文备注`
  - 每个字段提供占位符提示和示例
  - 故障码字段增加自动补全（根据历史记录）
  - 关键字段（如故障码）增加格式校验

```tsx
// 建议的输入结构
interface DiagnosisInput {
  faultCode: string;      // 必填，带格式校验（如 E-XXX / PROC-XXX）
  deviceType: string;     // 必填，下拉选择（航空装配工位/热处理设备/终检工位...）
  symptomText: string;    // 必填，多行文本
  contextNotes: string;    // 选填，补充说明
}
```

**7. 诊断结果可解释性增强**

- **现状**：用户看到诊断建议，但不明白为什么给出这个建议
- **问题**：AI 推理链路不透明，影响用户信任度
- **建议**：
  - 可视化展示 `evidence → diagnosis` 的推理链路
  - 每个建议旁边显示对应的证据来源（点击可展开）
  - 用颜色/图标标注支撑强度（🟢强支撑/🟡部分支撑/🔴弱支撑）

**8. 草稿暂存机制**

- **现状**：刷新页面或切换席位会导致输入内容丢失
- **问题**：用户输入了大量内容后，误操作导致重新填写
- **建议**：
  - 使用 `localStorage` 自动保存输入内容
  - 离开页面时提示 "是否保存草稿"
  - 支持 "从草稿恢复" 功能

#### 🟡 中优先级

**9. 操作引导系统**

- **现状**：`DiagnosisPage` 底部为静态 "流程说明" 文本
- **问题**：用户不知道当前该做什么
- **建议**：
  - 根据工作流状态动态显示操作引导：
    - 等待输入 → 显示 "请填写故障信息"
    - 分析中 → 显示 "AI 正在分析，请稍候..."
    - 待确认 → 显示 "请确认诊断结果或提出修改意见"
    - 已提交 → 显示 "工单已流转，等待审批"
  - 高风险结果增加醒目标识和确认提示

**10. 错误恢复引导**

- **现状**：`useYixiutongWorkspace.ts:273` 错误消息为 "诊断执行失败"
- **问题**：用户不知道如何恢复
- **建议**：
  - 区分错误类型：
    - 网络错误 → "网络连接失败，请检查网络后重试"
    - LLM 超时 → "AI 服务响应超时，请稍后重试"
    - 业务错误 → 显示具体原因（如 "知识库为空，请先上传文档"）
  - 每个错误类型提供具体的恢复操作指引

### 3.2 优化建议（用户体验方向）

**11. 执行过程可视化**

- **现状**：用户提交后只能等待，结果返回后才看到最终输出

- **建议**：
  - 显示工作流执行进度（如步骤卡片：📍路由 → 📍检索 → 📍诊断 → 📍校验 → ✅完成）
  - 每个节点显示执行状态（进行中✅/已完成⏳/失败❌）
  - 实时显示召回证据数量、置信度变化

```tsx
// 建议的执行进度展示
const workflowSteps = [
  { id: "route", label: "智能路由", status: "completed" },
  { id: "retrieve", label: "知识检索", status: "completed", detail: "召回 5 条证据" },
  { id: "diagnose", label: "诊断分析", status: "in_progress", detail: "正在生成建议..." },
  { id: "validate", label: "工单校验", status: "pending" },
```

- **现状**：用户需要主动刷新页面查看审批状态
- **问题**：审批状态变更无法及时知晓
- **建议**：
  - 利用已有的 `notifications.py` 消息通道配置
  - 审批通过/驳回时发送 Webhook 通知
  - 前端支持企业微信/飞书消息推送
  - 个人工作台显示待处理任务红点提醒

**13. 个人工作台（Dashboard）增强**

- **现状**：`DashboardPage` 仅显示数据总览
- **问题**：用户无法快速了解当日/本周工作任务
- **建议**：
  - 左侧显示快捷入口，右侧显示待办任务列表
  - "我的待审批" 工单列表（卡片形式）
  - "我的诊断历史" 最近诊断记录
  - 关键指标趋势图（如本周处理工单数、平均响应时长）

#### 🟡 中优先级

**14. 操作确认交互**

- **现状**：确认/驳回按钮无二次确认
- **问题**：误操作可能导致生产事故
- **建议**：
  - 审批确认增加二次弹窗："确认通过此工单？"
  - 高风险操作（驳回、升级 MRB）要求输入审批意见
  - 显示操作后果提示（如 "工单将流转至执行状态"）

**15. 前端状态同步优化**

- **现状**：`App.tsx:73-83` 场景预设应用存在竞态风险
- **问题**：直接通过 URL 访问不同席位时，`sceneType` 与 UI 不同步
- **建议**：
  - 将 `applyScenePreset` 逻辑移至 `workspace` 初始化阶段
  - 使用 React Context 统一管理场景状态
  - URL 参数作为 source of truth

## 四、工作流程分析与优化建议

### 4.1 工作流现存问题

| 问题 | 位置 | 描述 |
|------|------|------|
| 二次意见后死循环风险 | `graph.py:210-213, 373` | 二次意见后置信度仍低时，流程直接进入 draft，不重试 |
| 工单修复过于简单 | `graph.py:276-296` | 仅用占位符补充，内容无实质修复 |
| 审批决策无持久化 | `workflows.py:30-48` | 工单不存在时只记备注，不持久化 |

**代码问题1：二次意见逻辑缺陷**

```python
# graph.py:210-213
def _should_run_second_opinion(state: WorkflowState) -> str:
    if state["confidence"].overall_score < 60 and state.get("second_opinion_attempts", 0) < 1:
        return "second_opinion"
    return "draft"

# graph.py:373 - 二次意见后回到 trace 节点
graph.add_edge("second_opinion", "trace")
```

问题：二次意见后重新计算置信度，如果仍 <60，会再次进入 `second_opinion`，但 `second_opinion_attempts < 1` 限制为最多1次，最终仍会进入 `draft`。**建议**：明确告知用户 "置信度较低，建议人工复核"，而不是强制生成工单。

**代码问题2：工单修复仅用占位符**

```python
# graph.py:281-284
if "safety_notes" in issue_fields and not work_order.safety_notes:
    work_order.safety_notes = ["补充安全注意事项后方可提交。"]
if "step_items" in issue_fields and work_order.steps and not work_order.step_items:
    work_order.step_items = []
```

问题：校验失败时仅补充占位符文本，不修复实质内容。**建议**：根据诊断结果和证据自动生成安全备注和步骤项，或标记为 "需人工补充"。

### 4.2 优化建议（工作流程方向）

**16. 优化二次意见流程**

- 增加二次意见后的置信度评估说明
- 如果二次意见后置信度仍低，明确提示 "建议人工复核"，而不是强制生成工单
- 在响应中增加 `low_confidence_warning` 字段

**17. 工单修复机制增强**

- 修复失败时，允许用户手动补充而非仅占位符
- 提供 "复制参考" 功能，从同类历史工单复制安全备注

**18. 审批决策持久化**

- 确保所有审批操作都写入数据库
- 增加操作审计日志

- 建立统一的消息状态管理机制
- 使用 Toast 组件显示操作结果

## 九、总结

翼修通项目在航空维修场景的 AI 辅助诊断方向具有良好的架构设计，工作流清晰、可追溯性强。但在**行业痛点解决**、**易用性**、**用户体验**三个高优先级方向仍有较大优化空间：

1. **行业痛点**：检索能力、知识沉淀、自适应风险阈值是核心短板
2. **易用性**：输入引导、结果可解释性、草稿机制是提升用户留存的关键
3. **用户体验**：执行过程可视化、状态通知、操作引导是降低用户焦虑的核心

建议按照优先级逐步迭代，优先解决高优先级问题，确保核心业务流程的可用性和用户信任度。
